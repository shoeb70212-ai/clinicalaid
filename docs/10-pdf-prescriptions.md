## Overview

Prescriptions are generated client-side using `@react-pdf/renderer`.
No server-side PDF API. No external service. Zero data leaves the device during generation.
Generated PDF is a Blob — shared via Web Share API or printed via browser print.

---

## Why @react-pdf/renderer

- Generates vector PDF in browser memory — no screenshot approach (html2canvas breaks on mobile)
- Consistent A4 output regardless of device or browser
- No server round-trip — pure client compute
- Works offline once bundle is loaded

---

## NMC Mandatory Fields (every prescription must include)

1. Patient name (English)
2. Patient age
3. Date of prescription
4. Doctor's full name
5. Doctor's qualification (MBBS, MD, etc.)
6. Doctor's NMC registration number
7. Clinic name and address
8. Drug name (ENGLISH CAPITALS)
9. Dosage
10. Frequency (e.g., 1-0-1)
11. Duration (e.g., 5 days)
12. Doctor's signature

---

## Signature Lock Rule

Doctor's SVG signature is injected ONLY when `auth.uid()` matches the prescribing doctor.
If receptionist prints a duplicate copy:
- Signature block renders blank
- Footer text: "Digitally generated by clinic staff. Requires physical doctor signature."

This prevents the receptionist from printing forged prescriptions.
Doctor signature stored as encrypted blob in DB — decrypted only at render time on authenticated device.

## Patient Instructions — Language

Drug name: ALWAYS English capitals (NMC mandate)
Dosage instruction below drug name: patient's preferred language

```
PARACETAMOL 650mg
सुबह और शाम, खाने के बाद — 5 दिन
```

Source: `drug_instructions_i18n` table lookup.
Fallback chain: patient preferred language → Hindi → English.

## Schedule X Hard Block

Before PDF generation, validate all drugs:
```typescript
const hasBannedDrug = drugs.some(d => d.schedule === 'X');
if (hasBannedDrug) {
  throw new Error('Schedule X drugs cannot be included in digital prescriptions.');
}
```

Log every attempt. Cannot be bypassed. No exceptions.

## Sharing

```typescript
// Web Share API — uses doctor's own installed apps (WhatsApp, Email, Print)
// No WhatsApp Business API — avoids PII transmission to Meta servers
const blob = await generatePrescriptionPDF(prescriptionData);
const file = new File([blob], `prescription-${tokenNumber}.pdf`, { type: 'application/pdf' });

if (navigator.share && navigator.canShare({ files: [file] })) {
  await navigator.share({
    title: `Prescription — ${patientName}`,
    files: [file],
  });
} else {
  // Fallback: trigger browser download
  const url = URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url;
  a.download = `prescription-${tokenNumber}.pdf`;
  a.click();
}
```

## Storage (Path B — DPDP compliant)

Generated PDFs stored at: `{clinic_id}/prescriptions/{patient_id}/{visit_id}.pdf`
On patient DPDP erasure: these files are hard-deleted (see docs/07-storage.md).
DB clinical records (visits table) are retained for NMC compliance.
This is Path B — PDF physical file destroyed, database structure retained.

---

## DPDP Compliance for PDFs

PDFs contain burned-in PII (patient name, age, address).
This is necessary for pharmacy dispensing (Drugs and Cosmetics Rules 1945, Rule 65).
A prescription without patient name would be rejected at the pharmacy counter.

Therefore: on erasure request, PDF files are hard-deleted.
The clinical data (diagnosis, drugs, dosage) remains in the anonymized `visits` row.
The patient's name is gone from the system. The clinical record structure is preserved.
